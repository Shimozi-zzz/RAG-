# RAG 知识库问答系统 —— 测试文档

本文档用于验证 RAG 系统的检索召回、问答准确性及防幻觉能力。

---

## 一、RAG 系统概述

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索系统与大语言模型相结合的技术架构。它的核心思想是：在用户提出问题后，先从外部知识库中检索出相关的文档片段，然后将这些片段作为上下文提供给大语言模型，让模型基于真实资料生成回答，而不是仅依赖模型自身的训练记忆。

RAG 管道由三个核心阶段组成：

1. **文档入库阶段** — 用户上传文档后，系统通过文本切分器将长文档切割为若干文本块（Chunks），然后使用嵌入模型将每个文本块转化为向量，存入向量数据库。

2. **检索阶段** — 当用户提问时，系统将问题同样转化为向量，在向量数据库中通过余弦相似度计算找到语义最相近的 Top-K 个文本块。

3. **生成阶段** — 将检索到的文本块与用户问题拼接成 Prompt 模板，发送给大语言模型生成最终回答。

这种架构的核心优势在于：大模型不再依赖训练截止日期前的老旧知识，而是可以实时访问最新的外部知识库；同时，由于回答必须基于检索到的真实文档，大大降低了模型的幻觉（Hallucination）问题。

## 二、文本切分策略

文本切分是 RAG 系统中直接影响检索质量的关键环节。常用的切分器是 RecursiveCharacterTextSplitter，它按照一组分隔符的优先级顺序递归地将文档切分为指定大小的文本块。

### 2.1 Chunk Size（文本块大小）

Chunk Size 决定了每个文本块的字符数量。本系统默认设置为 500 字符。

- 如果 Chunk Size 设置过小（如 200 字符），文本块语义碎片化严重，一个完整的段落可能被切散到多个块中，导致检索时无法召回完整的上下文信息。

- 如果 Chunk Size 设置过大（如 1000 字符以上），单个文本块包含的噪声信息增多，检索精度下降，同时也会增加大模型输入的 Token 消耗。

- 经实验验证，300 到 500 字符是中文文本切分的最佳区间。

### 2.2 Chunk Overlap（重叠字符数）

Chunk Overlap 指定了相邻两个文本块之间的重叠字符数量，默认设置为 Chunk Size 的 10%，即 50 字符。

作用：防止关键信息恰好落在两个文本块的边界处而被切断。例如一个定义句可能前半句在一个块中、后半句在下一个块中，Overlap 确保边界附近的语义信息在至少两个块中都保持完整。

### 2.3 分隔符优先级

本系统使用的分隔符列表为：`["\n\n", "\n", "。", ".", " ", ""]`

- 优先在段落边界（双换行）处切分
- 其次在行边界（单换行）处切分
- 再次在句号处切分
- 最后在空格或字符级别切分

这种优先级设计使得文本块尽可能包含完整段落或完整句子，避免在句中断开。

## 三、Embedding 嵌入模型

### 3.1 什么是 Embedding

Embedding（嵌入）是将文本转化为高维空间中的数值向量的技术。相似语义的文本在向量空间中距离相近，不相似的文本距离较远。这种"语义坐标"能力是 RAG 检索的核心基础。

### 3.2 模型选择

本项目默认使用 **BAAI/bge-small-zh-v1.5** 模型，由北京智源研究院（BAAI）开发，专为中文语义检索优化。该模型特点：

- 约 100MB 大小，轻量级
- 可在 CPU 上高效运行，无需 GPU
- 支持向量归一化输出
- 中文语义理解能力强

备选模型 **BAAI/bge-large-zh-v1.5** 精度更高（约 1.3GB），适合答辩时进行精度对比演示。

### 3.3 首次运行注意事项

首次启动系统时，Embedding 模型需要从 HuggingFace 下载（约 100MB）。如果因为网络原因无法直接访问 HuggingFace，系统已默认配置 `HF_ENDPOINT=https://hf-mirror.com` 国内镜像站加速下载。

## 四、向量数据库

### 4.1 Chroma 简介

Chroma 是一个开源的向量数据库，专为 AI 应用设计。本项目选用 Chroma 的原因：

- **零配置**：pip install 后即可使用，无需额外安装 Docker 或服务器
- **自动持久化**：向量数据自动写入本地磁盘，重启后数据不丢失
- **SQLite 底层**：元数据存储于 SQLite，轻量且可靠
- **内置嵌入引擎**：可通过 ONNX Runtime 运行内置的 Embedding 模型

与其他向量库的对比：

| 数据库 | 配置难度 | 适用场景 |
|--------|---------|---------|
| Chroma | 零配置，pip 安装 | 课程设计、原型开发 |
| FAISS | 需手动管理持久化 | 研究实验 |
| Milvus | 需 Docker/集群 | 生产环境、大规模数据 |
| Pinecone | 云服务，按量付费 | 商业项目 |

### 4.2 核心操作

**数据入库：** 将切分好的文档块通过 Embedding 模型向量化后，调用 `add_documents_to_store()` 存入 Chroma。每个文档块存储其向量表示和原始文本内容。

**语义检索：** 调用 `similarity_search()` 方法，传入用户问题字符串和 Top-K 参数，Chroma 自动将问题向量化并与库中所有向量计算余弦相似度，返回最相似的 K 个文档块。

**数据管理：** 向量数据持久化在 `data/chroma_db/` 目录下。删除此目录即可清空整个知识库。

## 五、大语言模型集成

### 5.1 DeepSeek 模型

本项目使用 DeepSeek 作为生成模型，通过 OpenAI 兼容的 API 接口调用。

关键配置参数：

- `DEEPSEEK_MODEL` — 模型名称，推荐使用 `deepseek-chat`
- `temperature` — 温度参数，设为 0.3 以保证回答的事实性和一致性
- `base_url` — API 端点，使用 `https://api.deepseek.com/v1`

### 5.2 防幻觉机制

本项目在 Prompt 模板中设计了两层防幻觉机制：

**第一层：Prompt 硬约束**

在 System Prompt 中明确要求模型：
- 严格基于提供的背景知识回答
- 背景知识不足时，必须明确告知"根据现有知识库无法回答该问题"
- 禁止编造背景知识中不存在的信息

**第二层：来源溯源**

API 响应的 `sources` 字段会返回所有被引用的文档片段原文，用户可以交叉验证回答的真实性。

### 5.3 性能指标

| 指标 | 数值 |
|------|------|
| 单次 Embedding 耗时 | < 100ms（BGE-small，CPU） |
| 单次 API 请求耗时 | 1-3 秒（DeepSeek） |
| 文档入库速度 | 取决于 Embedding 模型和文档大小 |
| API 成本 | 约 1 元 / 百万 Token |

## 六、高级 RAG 技术展望

### 6.1 Self-Querying 检索

Self-Querying 允许 LLM 从用户自然语言问题中自动提取结构化查询条件（如"请查找 2024 年关于机器学习的 PDF 文档"），然后对向量库进行带元数据过滤的混合检索。

### 6.2 HyDE（假设文档嵌入）

HyDE 技术让 LLM 先生成一个假设性的回答文档，然后将这个假设文档进行 Embedding 并用于检索。由于 LLM 生成的文档在语义模式上与真实文档更接近，HyDE 能有效缩小问题与文档之间的语义鸿沟，显著提升检索召回率。

### 6.3 Rerank 重排序

Rerank 使用 Cross-Encoder 模型（如 bge-reranker-large）对 Top-K 检索结果进行精细的二次排序。与初次检索使用的 Bi-Encoder 不同，Cross-Encoder 将问题和候选文档同时输入模型进行联合编码，能更准确地评估相关性。实验表明，Rerank 可使 Top-1 准确率提升 10% 到 20%。

### 6.4 Multi-Vector 多向量检索

Multi-Vector 技术将每个文档块拆分为多个子向量，分别代表文档的不同语义维度（如主题、摘要、关键词等）。检索时可以同时匹配多个语义维度，提高对复杂查询的召回能力。

### 6.5 Agent 智能体

将 RAG 系统升级为 Agent，使其具备工具调用能力。例如：

- 用户问"知识库中 A 产品和 B 产品的价格差是多少？"
- Agent 自动拆解为：检索 A 价格 → 检索 B 价格 → 调用计算工具求差值 → 生成回答

## 七、系统部署架构

### 7.1 开发环境

开发者在 src/ 目录下使用系统 Python 运行，通过 dev.bat 启动，端口为 127.0.0.1:3000，启用 --reload 热重载。

### 7.2 发布包

通过 build_release.ps1 脚本构建发布包。发布包内置 Python 3.11.9 嵌入式环境，用户无需安装任何依赖，解压后双击 run.bat 即可启动。

### 7.3 三层分离设计

```
src/          → 开发者日常编辑（0.5MB）
dist/         → 发布包产物（1.3GB，不进 Git）
build_release.ps1 → 自动化构建脚本
```

这种设计解决了编辑器因扫描 900MB 便携 Python 环境而 OOM 崩溃的问题，同时保证了开发效率和发布质量。

## 八、常见问题与解决方案

### 8.1 API Key 配置

系统启动需要有效的 DeepSeek API Key。申请地址：https://platform.deepseek.com

配置方式：将 .env.example 重命名为 .env，编辑文件，将 DEEPSEEK_API_KEY 字段替换为真实 Key。

### 8.2 端口被占用

Windows 的 Hyper-V、WSL 或 IIS 可能会预留一段端口范围。当 run.bat 启动失败时：

1. 脚本会自动从 8888 切换到 9999
2. 如果两个都失败，编辑 .env 文件，修改 PORT=3000
3. 如果仍然失败，可尝试 3001、5000、8080 等端口

可使用以下命令检查哪些端口被系统预留：
```
netsh interface ipv4 show excludedportrange protocol=tcp
```

### 8.3 模型下载问题

首次运行需要从 HuggingFace 下载 Embedding 模型（约 100MB）。如果下载失败：

- 确认 .env 中 HF_ENDPOINT 配置正确（默认已设为国内镜像）
- 关闭代理软件或配置代理规则
- 检查网络连接

模型下载成功后，后续启动无需再次下载，启动速度秒级。

### 8.4 文件格式支持

系统支持上传以下格式的知识库文档：

- PDF 文件（.pdf）
- Markdown 文件（.md）
- 纯文本文件（.txt）
