# 检索增强生成（RAG）技术白皮书

## 一、什么是 RAG

RAG 全称为 **Retrieval-Augmented Generation**（检索增强生成），是一种将信息检索与大语言模型生成能力相结合的技术架构。

传统大语言模型存在两个核心痛点：
1. **知识截止日期限制**：模型训练数据有截止时间，无法回答训练后发生的事件。
2. **幻觉问题**：模型可能编造看似合理但事实错误的内容。

RAG 通过在生成回答之前，先从外部知识库中检索相关文档片段作为上下文，再将检索结果与用户问题一起送入大模型，有效缓解了上述问题。

## 二、RAG 的核心流程

RAG 系统分为两个阶段：

### 离线索引阶段（Indexing）
1. **文档加载**：从 PDF、Markdown、TXT 等格式加载知识库文档。
2. **文本切分**：使用 TextSplitter 将长文档切分为适合检索的小块（Chunk），典型 Chunk 大小为 300~500 字符。
3. **向量化**：使用 Embedding 模型（如 BGE、text-embedding-3）将每个 Chunk 转换为向量。
4. **存储**：将向量存入向量数据库（如 Chroma、FAISS、Milvus）。

### 在线检索阶段（Retrieval & Generation）
1. **用户提问**：接收用户自然语言问题。
2. **问题向量化**：将问题转换为向量。
3. **相似度检索**：在向量库中检索 Top-K 个最相似的文档片段。
4. **Prompt 组装**：将检索到的文档片段作为上下文，与用户问题一起构造提示词。
5. **答案生成**：大模型基于上下文生成回答，并标注信息来源。

## 三、关键技术参数

### Chunk Size（块大小）
- 太小（<200）：语义碎片化，丢失上下文。
- 太大（>1000）：检索精度下降，噪声增加。
- 推荐：中文文本 300~500 字符。

### Chunk Overlap（块重叠）
- 在相邻块之间保留一定重叠，避免关键信息被切断。
- 典型值：Chunk Size 的 10%~20%。

### Top-K
- 检索返回的文档块数量。
- K 太小：信息不充分。
- K 太大：引入噪声，Token 消耗增加。
- 推荐值：3~7。

### Embedding 模型选择
- 中文场景推荐：BAAI/bge-large-zh-v1.5（精度高）或 BAAI/bge-small-zh-v1.5（速度快）。
- 多语言场景：text-embedding-3-small、sentence-transformers/all-MiniLM-L6-v2。

## 四、高级 RAG 技术

### Self-Querying
让 LLM 自动从用户问题中提取元数据过滤条件（如时间范围、文档类型），结合向量相似度检索和结构化过滤。

### Multi-Vector Retrieval
为每个文档块生成多个不同粒度的向量（如摘要向量 + 正文向量），提升召回率。

### HyDE（Hypothetical Document Embeddings）
先用 LLM 生成一个假设性答案，再用该答案向量进行检索。适用于用户问题简短、信息量不足的场景。

### Rerank
在初步检索后，使用 Cross-Encoder 模型对结果进行重排序，提升 Top-1 准确率。常用模型：bge-reranker-large。

## 五、RAG 的典型应用场景

- **企业知识库问答**：内部文档、SOP、产品手册的智能检索与问答。
- **法律文书分析**：法规库、判例库的快速查阅与引用。
- **医疗辅助**：基于医学文献的临床决策支持。
- **教育辅导**：教材/论文库的自动答疑。
- **客服系统**：基于产品知识库的自动应答。

本项目实现了上述 RAG 核心架构，具体选型如下：

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | FastAPI | 高性能异步框架，自带 Swagger 文档 |
| LLM | DeepSeek-Chat | 高性价比，OpenAI 兼容接口 |
| Embedding | BAAI/bge-small-zh-v1.5 | 本地运行，零费用，中文支持好 |
| 向量库 | Chroma | 轻量、自动持久化，与 LangChain 深度集成 |
| 编排框架 | LangChain | 统一的文档加载/切分/检索/LLM 抽象层 |
